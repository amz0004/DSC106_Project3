Project Write-Up

DESIGN PROCESS

Our overall design goal was to create an interactive visualization that enables users to explore patterns in U.S. wildfire activity across space, time and intensity. Since the fire data is inherently spatial and highly dense, we try to centered our design around a geographic map view paired with interactive filtering tools that allow users to reveal any patterns without getting overwhelmed by all the points of data.

Visual Encodings

Fire Dot Color and Size

To represent the fire’s brightness, we used two visual encodings. One being the color progressing from red (lower brightness) to yellow (higher brightness), and the other being the circle size, where the larger dots represent brighter fires. In the beginning, we experimented with a purple to red to yellow color scale, but the darker purple values' brightness levels blended in too much with the background (dark grey). The final red to yellow palette produced clearer contrast and aligned intuitively with “heat” as a visual metaphor.

Background and Map Styling

We used a dark grey U.S. map over a slightly lighter grey page background to give it similar mood and color visual of smoke and ash. This also ensured a high contrast between the map and the fire markers.

Interaction Techniques

Hoverable Tooltips

Since color and size alone aren’t the most precise indicators of brightness, we added tooltips showing the numeric brightness, date and time, fire detection mode, state, and coordinates. This provides the users details immediately without cluttering the map visually.

Minimum Brightness Slider

The dataset contains thousands of fire detections, which overwhelmed the visualization and caused long loading times if it was shown unfiltered. A minimum-brightness slider allows users to declutter the map and focus on more significant fires. We also linked the slider’s color to the brightness scale, so the UI would be more intuitive since the slider’s appearance matches the actual data being filtered.

Season Filter

A season filter was created since environmental conditions influencing fire activity (humidity, wind, dry vegetation) vary strongly by season. We considered first letting users filter by month, but 12 checkboxes started to clutter the UI. Grouping months into 4 seasons preserved more insight while keeping the interface clean.

Dynamic Monthly Brightness Line Graph

During the early EDA, it was revealed that there were clear seasonal patterns in average brightness. To support more insight, we added a dynamic line graph showing the average fire brightness per month, updated in real time based on the minimum brightness threshold, the selected states (or entire US when none are selected), and selected seasons. The graph always shows all 12 months for easy comparison, but highlights the months included in the selected seasons.

Clickable State Selection

Many users have personal ties to specific states, so we implemented a selection interaction for focusing on individual states or multi-state regions. When adding the multi-selection, it presented a zooming challenge. It would start auto-zooming immediately after selecting a single state which prevented users from selecting additional states. We solved this by adding a Zoom button that’s allowing the user to select multiple states first and then zoom in. A Reset button then clears all selections at once for some convenience.

Dynamic Titles and Subtitles

We wanted to avoid confusion about which subset of the data is being visualized, so the title is automatically updated to reflect selected seasons and the associated date range.

Alternatives Considered

When we started, we first prototyped an animation-based time line showing fire activity frame by frame over time. While the graph was visually engaging, this approach limited the control the user had and made exploration linear. We pivoted to a filter-based approach to let the users to enable deeper, more flexible investigations. We also debated shortening the time range of the data instead of filtering by brightness. Doing that though would hide important seasonal patterns. In the end, we decided to remove fires below 325 K because they’re low-intensity events and were a large chunk of the dataset.

DEVELOPMENT PROCESS

Team Workflow

We began by going over several initial ideas, such as an animated fire timeline or a simple static map. Through discussion and some prototyping, we narrowed the project down to an interactive map visualization that paired filtering tools with a dynamic brightness chart. We documented all decisions, both accepted and removed, in a shared Google Doc.

Division of Labor

Based on our own strengths, we divided the responsibility:
• Amy: Initial visualization planning, base map setup, state boundaries, fire plotting, data truncation, brush/selection experiments.
• Vanessa: Data collection, tooltip implementation, brightness slider, major logic for the dynamic line graph, and selection-related interactions.
• Bryan: Research, season filtering logic, dynamic subtitles/title system, and write-up development.

Time Spent

Across planning, prototyping and implementation, we estimated the time spent was approximately 30 total people-hours.

Most Time-Consuming Aspects

Some of the largest time investments parts for our group was settling on a final concept, implementing a robust state/season selection, debugging brush interactions (which ultimately led to click selection), and managing the dataset size and visualization performance.

Challenges and Iterations

We constantly and repeatedly refined interactions as we felt clarity and user exploration was most important. For example, we saw that months vs seasons was a significant design tradeoff -- the auto-zooming became disruptive till we introduced manual zoom controls. Additionally, the brush selection was explored for 2 to 3 hours but then was abandaoned due to performance and usability issues. All these iterations shaped our design that’s both functional and intuitive.
